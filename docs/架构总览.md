# OpenCode 轻量级 Subagent 插件架构方案 v2

## 1. 项目定义

本项目是一个面向 OpenCode 的轻量级 subagent 插件体系。

它参考 OMO 的交互形态与部分运行时能力，但不复刻其重运行时，而是采用 **config-first, plugin-second** 的路线，并引入一个**受控的轻量运行时协调层**。

该体系保留三类用户可见入口、固定角色骨架、artifact-first 状态模型与显式审批门；同时允许引入以下能力的简化版本：

- simplified category routing
- configurable child orchestrator
- limited bundled MCP
- minimal managers/hooks
- bounded background coordination

这些能力只能作为既有主流程的增强层存在，不能演化为第二套隐藏运行时。

## 2. 项目目标与非目标

### 2.1 项目目标

1. 提供三个用户可见入口：执行、规划、深度规划。
2. 保留 7 个核心角色，并明确 primary 与 internal-only 的边界。
3. 保持执行域与规划域的清晰分工。
4. 保留 Explore、Librarian、Review 作为共享内部能力，而不是用户入口。
5. 保留 OMO 风格的外在行为：**文件产物 + 对话摘要 + 显式审批门**。
6. 保持 artifact-first 状态所有权，避免把 session/context 作为真相源。
7. 允许有限运行时增强，但保持系统可解释、可降级、可验证。
8. 在 V1 即定义最小测试面与 TDD 验证架构。

### 2.2 非目标

1. 不复刻 OMO 的完整重运行时。
2. 不引入动态无限扩张的 category taxonomy。
3. 不允许深层 orchestrator 树。
4. 不把 bundled MCP 做成开放式能力平台。
5. 不把 managers/hooks 做成通用事件总线或中间件框架。
6. 不把后台协调做成持久自治 worker 系统。
7. 不把插件定义为唯一正确性来源。
8. 不设计函数级、类级或模块级实现细节。

## 3. 设计原则

1. **用户只选模式，不操作内部能力**  
   用户看到的是执行、规划、深度规划，而不是 Explore/Librarian/Review/MCP/hook 等内部能力。

2. **执行域与规划域分离**  
   执行域负责“把事做出来”，规划域负责“把事想清楚”。

3. **共享能力不抢控制权**  
   Explore、Librarian、Review 只提供能力，不拥有主流程控制权。

4. **config-first, plugin-second**  
   优先使用 OpenCode 原生 agent/config/permission/skills 能力。插件只做薄增强。

5. **bounded-lite runtime**  
   允许运行时增强，但增强必须受控、显式、可降级，不能成为第二运行时。

6. **artifact-first**  
   artifacts are the source of truth。child sessions are disposable。compaction 只能改善体验，不能承载正确性。

7. **显式审批门**  
   Review 只能审，不能静默改。关键关卡必须显式可见。

8. **失败局部化**  
   路由、MCP、hook、子编排、后台任务的失败必须局部吸收，不能蔓延成全局复杂性。

## 4. 用户可见入口模式

### 4.1 执行模式（Command Lead）

适用：

- 简单任务
- 已明确的复杂任务
- 用户希望直接推进

行为：

- 判断是否需要探索、外部检索、骨架规划、详细规划
- 必要时调用 Plan Builder normalize mode
- 进入执行移交与结果审查链路

### 4.2 规划模式（Plan Builder）

适用：

- 需求尚不稳定
- 功能范围、架构边界、技术方向还不清楚
- 用户明确希望先规划

行为：

- 默认以 discussion mode 工作
- 产出结构化计划骨架
- 可在骨架稳定后结束，或升级到深度规划

### 4.3 深度规划模式（Power Plan Builder）

适用：

- 项目复杂
- 需要执行级蓝图
- 需要更严格的计划审查

行为：

- 以稳定 skeleton 为输入
- 输出详细计划
- 自动进入 plan review

### 4.4 模式升级规则

Power Plan Builder 不应成为默认总会介入的第二规划器。仅在以下情况介入：

1. 用户显式选择深度规划；
2. Plan Builder 已产出稳定骨架，但复杂度超出普通规划；
3. Command Lead 在高风险或高耦合任务前要求执行级方案。

## 5. 角色拓扑与注册模型

### 5.1 Primary 角色

- 主编排 / Command Lead
- Plan Builder
- Power Plan Builder

### 5.2 Internal-only/Subagent 角色

- 子编排 / Task Lead
- Explore
- Librarian
- Review

### 5.3 角色拓扑

```text
User
 ├── Command Lead
 │    ├── Task Lead
 │    ├── Plan Builder
 │    ├── Power Plan Builder
 │    ├── Explore
 │    ├── Librarian
 │    └── Review
 ├── Plan Builder
 │    ├── Explore
 │    ├── Librarian
 │    └── Review
 └── Power Plan Builder
      ├── Explore
      ├── Librarian
      └── Review
```

### 5.4 委派规则

1. 只有主编排与子编排拥有**自由委派权**，可以根据任务需要决定是否调用其他角色。
2. Plan Builder 与 Power Plan Builder 不拥有通用编排权，但可以在其固定职责范围内调用共享能力，如 Explore、Librarian、Review。
3. Explore、Librarian、Review 都不是自由委派者，不能继续展开新的编排链。
4. 子编排最大嵌套深度默认是 `1`。
5. 不允许形成深层 orchestrator 树。

## 6. 角色合同

### 6.1 主编排 / Command Lead

- **职责**：接收执行类请求，拥有总路由、审批所有权、状态推进权。
- **输入**：用户请求、已有 artifacts、探索/检索摘要。
- **输出**：路由决定、执行移交、审批请求、结果摘要。
- **退出条件**：任务交付、阻塞于审批、或显式转入其他模式。
- **不得做**：把 Explore/Review 变成主流程控制者。

### 6.2 子编排 / Task Lead

- **职责**：处理局部复杂度，组织单任务范围内的内部能力。
- **输入**：主编排下发的单任务目标与边界。
- **输出**：局部任务摘要、受控 artifact 更新建议。
- **退出条件**：单任务完成或失败后立即结束。
- **不得做**：生成更深子编排树；拥有最终审批权。

### 6.3 Plan Builder

#### discussion mode

- **触发**：用户明确要求先规划，或任务明显模糊。
- **行为**：与用户多轮讨论。
- **输出**：目标、范围、边界、假设、风险清晰的计划骨架。

#### normalize mode

- **触发**：主编排判断上下文已基本足够。
- **行为**：不打断用户，只归一化收束。
- **输出**：结构化 skeleton。
- **不得做**：演化为第三个独立 planner。

### 6.4 Power Plan Builder

- **职责**：把稳定骨架展开成执行级详细计划。
- **输入**：稳定 skeleton。
- **输出**：详细执行蓝图、阶段、依赖、审批点、验证点。
- **不得做**：在骨架未稳定时强行细化。

### 6.5 Explore

- **职责**：本地上下文、代码、文件、结构探索。
- **输出**：事实摘要、相关对象清单。
- **不得做**：替代上层做最终决策。

### 6.6 Librarian

- **职责**：外部文档、官方资料、开源案例检索。
- **输出**：外部参考摘要、来源、适用性判断。
- **不得做**：替代 Plan Builder 或 Review 输出最终结论。

### 6.7 Review

- **职责**：审计划、审执行结果、审边界与保真。
- **输出**：通过 / 驳回 / 升级到用户裁决。
- **不得做**：静默改写计划或结果。

## 7. 权限与边界模型

### 7.1 双层边界

1. **permission.task as primary**  
   原生权限系统是主边界。
2. **plugin guardrail as secondary**  
   插件守卫是次级防护，不是唯一安全边界。

### 7.2 可见性边界

- Primary roles 可直达。
- Internal-only roles 默认隐藏。
- `direct user invocation` 是工作流 caveat，不作为绝对安全前提。

### 7.3 工具边界

- 自定义工具必须 namespaced。
- 不允许工具名无策略碰撞。
- 工具边界服务于“防越权”和“防隐藏直达”。

### 7.4 启动边界

- 初始化必须避免 client API calls during init。
- 插件启动必须轻量、安全、可回退。

## 8. 框架分层

```text
Config Layer
Runtime Coordination Layer
Plugin Layer
Artifact Layer
```

### 8.1 Config Layer

负责：

- 角色定义
- 模式暴露
- permissions
- prompts / skills
- policy defaults
- route declarations

### 8.2 Runtime Coordination Layer

这是新增的轻量协调层，但不是第二运行时。它只允许受控承载以下能力：

- category routing
- child orchestrator slot
- minimal managers
- bounded background coordination
- minimal lifecycle hooks

### 8.3 Plugin Layer

只做薄增强：

- secondary guardrail
- bundled MCP 接入封装
- artifact 索引
- hook 注册
- 启动安全辅助逻辑

### 8.4 Artifact Layer

负责：

- plan skeleton
- detailed plan
- approval state
- execution summary
- child task return summary
- review result

## 9. Runtime Coordination Layer 正式定义

### 9.1 Category Routing

允许使用 category routing，但必须满足：

- category 集合小且稳定
- 只映射到既有模式、角色、审批强度
- 路由应声明式、静态可读、可回退
- 不允许 router 套 router
- 不允许其演化成复杂策略引擎

这些 category 是**内部路由意图**，不是新增的用户模式。

建议的初始类别：

- `execution`
- `planning`
- `deep-planning`
- `explore`
- `librarian`
- `review`

### 9.2 Configurable Child Orchestrator

允许子编排可配置，但必须满足：

- 主编排仍是唯一控制平面所有者
- 子编排只在单任务范围内工作
- 子编排不能定义审批语义
- 子编排不能定义 artifact 所有权
- 子编排最大嵌套深度默认仍为 `1`

### 9.3 Limited Bundled MCP

允许 bundled MCP，但必须是受限捆绑：

- 必须显式声明包含哪些 MCP
- 必须区分 core bundled MCP 与 optional bundled MCP
- MCP 失败时必须局部降级
- 不允许自动发现、自动扩容、任意接入
- 它是受控能力集，不是开放生态位

### 9.4 Minimal Managers

允许 manager，但只保留最小集合：

- `routing manager`
- `artifact coordination manager`
- `review gate manager`
- `runtime safety manager`

这些 manager 只是 Runtime Coordination Layer 内的协调职责，不是新的框架层，也不是独立状态所有者。它们只能围绕既有主流程工作，不能引入新的主状态机。

### 9.5 Minimal Hooks

允许 hook，但只允许少量关键节点：

- `before dispatch`
- `after dispatch`
- `before approval`
- `after review`
- `on background task settled`

约束：

- hook 不得触发隐藏二次编排
- hook 不得改写主状态机
- hook 不得形成无限链式调用

### 9.6 Bounded Background Coordination

允许简化版后台协调，但仅限于：

- 任务提交
- 状态跟踪
- 结果汇总
- 超时处理
- 显式取消

不允许：

- 持久自治 worker
- 不可见跨 session 长期存活任务
- 隐式无限重试
- 自我扩张式任务网络

后台任务的提交、跟踪、超时、取消属于**辅助观测与协调信息**，不构成第二套 canonical state model。

## 10. 工作流与状态机

### 10.1 主流程

```text
用户请求
→ 主编排分流
→ Explore / Librarian（如需要）
→ Plan Builder（如需要，discussion / normalize）
→ 稳定骨架（如需要）
→ Power Plan Builder（如需要）
→ plan review（如需要）
→ 执行移交
→ execution-output review（如需要）
→ 用户审批（仅在需要显式批准时）
→ 结果交付
```

简单执行任务可以跳过规划链路；只有在任务复杂度、风险或边界不确定性达到阈值时，才进入 Plan Builder、Power Plan Builder 或显式审批门。

### 10.2 Canonical State Model

- `draft`
- `needs-approval`
- `approved`
- `executing`
- `reviewed`

### 10.3 状态定义

- **draft**：计划或结果仍在形成中
- **needs-approval**：达到显式审批门
- **approved**：关键关卡获批
- **executing**：执行或执行移交中
- **reviewed**：审查完成并有明确结论

### 10.4 状态转移触发器

- `draft → needs-approval`：形成稳定 skeleton、详细计划或待交付结果
- `needs-approval → approved`：用户或流程明确批准
- `approved → executing`：已批准对象被移交执行
- `executing → reviewed`：执行结果进入 Review 并完成审查
- `reviewed → needs-approval`：需用户裁决或显式确认
- `reviewed → draft`：被驳回且需重产出
- `reviewed → approved`：审查通过并可继续推进或交付

## 11. Artifact 模型与所有权

### 11.1 Artifact 类型

- 计划骨架
- 详细计划
- 审批状态
- 执行摘要
- 子任务返回摘要
- Review 结论

### 11.2 所有权

- 主编排拥有主流程 artifact 的协调所有权
- Plan Builder / Power Plan Builder 拥有其计划产物的产出责任
- Review 拥有审查结论产出责任，但没有静默改写权
- 子编排只拥有局部任务摘要责任

### 11.3 正确性规则

- artifacts are the source of truth
- child sessions are disposable
- session/context 只是运行态缓存
- compaction 不能替代 artifact 记录

## 12. 审批与 Review 模型

### 12.1 两类审核面

1. `plan review`
2. `execution-output review`

### 12.2 Review 输出

- 通过
- 驳回并说明原因
- 升级到用户裁决

### 12.3 Approval Gates

- approval gates 必须可见且显式
- 用户必须知道：审批对象、审批原因、可批准内容、拒绝后的影响
- 审批门不能隐藏在内部黑箱里

### 12.4 驳回循环规则

- after 2 rejection cycles, escalate to user
- 不允许无限 reject/rewrite 循环
- Review 只能审，不能静默重写

## 13. Full Mode 与 Degraded Mode

### 13.1 Full Mode

组成：

- config + thin plugin + runtime coordination enhancements

能力：

- 更强的守卫
- 更好的 artifact 索引
- 更完整的 MCP 封装能力
- 更好的后台任务观测与审批体验

### 13.2 Degraded Mode

组成：

- config-only 或 config + 最小 plugin 壳

保留：

- 3 个用户入口
- 7 角色结构
- 基本模式分流
- 基础 planning / explore / review 链路

损失：

- 部分 plugin-assisted 严格性
- 部分 MCP 打包能力
- 部分后台协调易用性
- 部分 hook/manager 增强

### 13.3 降级原则

- degraded mode 不要求 identical behavior in degraded mode
- 降级目标是“仍可用”，不是“完全等价”

## 14. 扩展模型

### 14.1 新角色扩展条件

新增角色必须回答：

1. 它是用户入口还是内部能力？
2. 它的输入/输出合同是什么？
3. 它是否真的不能被现有 7 角色吸收？
4. 它是否破坏轻量运行时边界？

### 14.2 新类别扩展条件

新增 category 必须证明：

- 现有 category 无法表达
- 不会引入新主流程原语
- 不会显著增加路由复杂度

### 14.3 新 MCP 扩展条件

新增 bundled MCP 必须证明：

- 高价值且高复用
- 可清晰声明为 core 或 optional
- 失败能局部降级
- 不扩大为开放生态接口

### 14.4 扩展默认立场

- V1 不鼓励增加角色数
- 优先扩展现有角色合同
- 扩展必须服从 bounded-lite runtime 原则

## 15. 运行时不变量

无论是否启用 routing、bundled MCP、hooks、background coordination，都必须保持：

1. **单一可见运行模型**  
   用户只看到 3 个模式，而不是动态代理图。

2. **主编排最高控制权**  
   最终路由、审批、artifact 提交、状态推进都由主编排拥有。

3. **单一主状态机**  
   routing、hooks、background coordination 都不能各自维护独立主状态机。

4. **artifact-first**  
   状态正确性必须落在 artifacts 上。

5. **无隐藏自治**  
   所有后台工作必须可枚举、可观察、可取消。

6. **失败局部化**  
   MCP、hook、manager、子编排的失败必须回退到基础流程，而不是拖垮系统。

## 16. 验证架构与成功标准

### 16.1 V1 最小验证面

- `config validity`
- `permission matrix`
- `plugin load safety`
- Plan Builder 双模式行为
- `review-gate`
- category routing 的声明一致性
- bundled MCP 的局部降级行为
- background task 的可枚举与可取消性

### 16.2 验证原则

- 不能只依赖手工测试
- 不一开始引入庞大 E2E harness
- 验证面必须服务“轻量且受控”的定位

### 16.3 可判定成功标准

- 用户能用一句话描述系统：**“主编排基于固定角色骨架进行分流，必要时调用受控扩展能力。”**
- 不新增第四个用户可见模式
- 仍然只有一套主状态机
- category 数量保持小规模且稳定
- manager 数量保持在最小集合
- hook 点保持在少量关键节点
- bundled MCP 失败时不阻塞基础主流程
- 关闭可选扩展后，系统仍退回基础设计
- 所有后台任务都能被列出、追踪、取消
- 配置面仍能保持在一份紧凑参考中，而不是框架手册

## 17. 最终验收波次

### F1. 方案一致性审计

检查：

- 7 角色命名与职责是否一致
- primary / internal-only 划分是否一致
- 5 个 canonical states 是否一致
- artifact-first 原则是否前后一致

### F2. 架构质量审阅

检查：

- 是否引入重运行时依赖
- 是否让共享能力抢控制权
- 是否让插件变成唯一正确性来源
- 是否出现深层 orchestrator 树

### F3. 验证设计审阅

检查：

- 核心设计断言是否都有对应验证面
- approval / permission / degraded mode / background coordination 是否可验证
- 是否只剩口号，没有落点

### F4. 范围保真检查

检查：

- 是否越界写入实现细节
- 是否偷偷恢复成 OMO 重能力
- 是否把设计稿写成实现分解稿

## 18. 风险声明

本方案最容易失控的地方不是角色数量，而是运行时增强能力的扩张：

- **control-plane drift**：routing、child orchestrator、hooks 都在做决策
- **extension creep**：简化版 managers/hooks 长成框架
- **hidden async**：后台协调演化为不可见调度系统
- **MCP surface sprawl**：bundled MCP 面积持续膨胀

因此必须坚持：**这些能力是受控增强，不是默认无限扩展点。**

## 19. 一句话定义

**A bounded-lite OpenCode subagent architecture with three visible modes, seven fixed roles, artifact-first state ownership, explicit approval gates, and strictly constrained runtime enhancements.**
