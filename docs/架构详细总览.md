# OpenCode 轻量多 Agent 编排架构 v1

本仓库是 OpenCode 的轻量全局编排插件，不是通用应用框架。核心交付物是 `opencode.json`、`.opencode/agents/**`、`.opencode/plugins/**`、`.opencode/lib/**` 与 `scripts/install.mjs`。

## 1. 固定角色拓扑

用户只看到 3 个入口：

- `command-lead`：执行入口。
- `plan-builder`：规划入口，强模型，Plan Review 可选。
- `deep-plan-builder`：深度规划入口，产出适合低强度执行模型交接的详细计划，Plan Review 强制。

隐藏角色固定为 5 个：

- `task-lead`：单个受限任务执行编排。
- `explore`：本地只读探索。
- `librarian`：外部资料和开源参考检索。
- `plan-review`：计划产物审查。
- `result-review`：用户可选择调用的结果审查，只审 Command Lead 拥有的执行摘要/最终整合结果。

OpenCode 内置 `build` 与 `plan` 仍通过隐藏禁用覆盖保留，避免用户绕过本插件入口。

## 2. 委派矩阵

```text
Command Lead       -> Task Lead, Plan Builder, Deep Plan Builder, Explore, Librarian, Plan Review, Result Review
Plan Builder       -> Explore, Librarian, Plan Review
Deep Plan Builder  -> Explore, Librarian, Plan Review
Task Lead          -> Explore, Librarian
Plan Review        -> Explore
Result Review      -> Explore
Explore            -> none
Librarian          -> none
```

`MAX_CHILD_ORCHESTRATOR_DEPTH = 1`。任何角色、hook、manager、后台协调都不能创建第二层 orchestrator。

## 3. 执行入口

Command Lead 拥有总路由、审批、artifact 提交、状态推进和最终验收。

简单任务可直接执行，不强制进入 Result Review。Result Review 是用户可选择调用的独立结果审查，只审 Command Lead 的 `execution-summary` 或最终整合结果。当路由阈值选择 Plan Builder 时，应先收集必要的 Explore/Librarian 事实，再把结构化 payload 传给 Plan Builder：

```yaml
payload:
  user_request: <用户原话>
  upstream_evidence:
    explore: []
    librarian: []
  constraints: []
```

payload 是结构化事实包，不是自然语言摘要。下游默认先消费 payload，只有缺少阻塞事实时才补充 Explore/Librarian，避免重复拉取同一信息。

任何有委派权的角色派遣 subagent 时都必须使用显式字段模板，不使用隐藏 initiator marker。Command Lead 是主派遣者，Plan Builder、Deep Plan Builder、Task Lead、Plan Review 和 Result Review 在调用其允许的下游角色时也使用同一模板：

```text
TASK:
<单个有边界的任务>

EXPECTED OUTCOME:
1. <必须产出的内容>
2. <必须产出的内容>

ROLE:
<task-lead|explore|librarian|plan-review|result-review>

SCOPE:
<允许检查/修改的文件、目录、模块、worktree、branch 或行为边界>

UPSTREAM EVIDENCE:
- explore: []
- librarian: []
- constraints: []

REQUIRED TOOLS:
<允许工具和命令类别；bash 尽量限制到安全命令族>

MUST DO:
- 维护本角色本地 todo。
- 先消费 upstream evidence。
- 用限定范围内的来源验证结论。

MUST NOT DO:
- 不越过 SCOPE。
- 不假设未检查的代码或文档现状。
- 不做 whole-repo 无边界搜索。

CONTEXT:
<用户原话、worktree 状态、已知决策和风险>

DELIVERABLE FORMAT:
<要求的小节、schema 或 artifact 格式>

FAILURE RETURN:
progress: <当前进度>
blocker: <具体阻塞>
artifacts: [<已有产出>]
recoverability: recoverable|partial|blocked
```

Task Lead 派遣必须包含 plan node id、`depends_on`、`attributes` 和 `deliverable`。只读角色必须在 `MUST NOT DO` 中明确禁止编辑和实现。Result Review 的审查对象只能是 Command Lead 的 `execution-summary` 或最终整合结果。

## 4. 规划入口

Plan Builder 是强模型规划者，面向需求澄清和可执行计划骨架。它需要多轮明确用户需求、边界和验收标准，不能替用户补产品决策。Plan Review 是可选项。

Deep Plan Builder 是深度规划者，负责输出适合低强度执行模型交接的执行级详细计划，并用强制 Plan Review 控制交接风险。它同样需要先明确需求和边界，再输出执行级计划文件。

## 5. 计划文件 Schema

可执行核心必须包含：

```yaml
plan:
  subtasks:
    - id: <unique>
      depends_on: [id, ...]
      attributes: [code, multimodal]
      deliverable: <Command Lead / 可选 Result Review 的验收依据>
      description: <做什么>
```

规则：

- `depends_on` 必填，空数组也必须写。
- `attributes` 是 tag set，用于配置化模型/执行 profile 派发。
- Command Lead 根据 `depends_on` 构建 DAG。
- 并发上限保持在 3-5。
- 派发映射必须配置化，不把具体模型写死在计划解析逻辑里。

## 6. Review 规则

Plan Review 只审 Plan Builder / Deep Plan Builder 的计划产物。Result Review 只审 Command Lead 拥有的 `execution-summary` 或最终整合结果，不审 Task Lead 的 `child-task-return-summary`。Result Review 不强制调用，用户可以在需要独立结果核查时选择调用。

统一输出：

```yaml
decision: pass|reject|escalate
severity: minor|major
findings:
  - location: <文件/行/函数/section>
    issue: <具体问题>
    pass_criteria: <可验证通过标准>
```

Review 侧拥有 severity 判定权。作者不能把 reviewer 的 `major` 自行降级成 `minor`。

迭代规则：

- `pass`：结束。
- `reject + minor`：作者直接修，不再强制重审。
- `reject + major`：作者修后重新 Review。
- 重大问题最多允许 2 次修订迭代；第三次重大失败升级给 Command Lead 或用户。
- 说不清楚怎么改的反馈直接视为 `major`，不能让作者猜。

## 7. Todo 纪律

所有角色都需要参考 OpenCode 的工作方式维护自己的本地 todo 列表：

- 多步工作开始时先列出本角色 todo。
- 执行中随进展更新 todo 状态。
- 结束前确认本角色 todo 已收敛或明确剩余阻塞。
- todo 是角色私有工作记忆，不是跨角色控制平面，不能替代 artifact、payload 或 canonical state。

## 8. 失败处理

Task Lead 失败时必须返回：

```yaml
progress: <当前进度>
blocker: <阻塞点>
artifacts: [<已有产出>]
recoverability: recoverable|partial|blocked
```

Command Lead 根据 DAG 判断影响：

- 独立叶子失败：记录失败，继续不受影响的任务，最后汇总。
- 关键路径失败：停止下游，升级给用户。

## 9. Canonical State

主状态机固定为：

```text
draft -> needs-approval -> approved -> executing -> reviewed
reviewed -> needs-approval
reviewed -> draft
reviewed -> approved
```

只有 Command Lead 可以推进 canonical state。Review 只产出审查结论，不能静默改写 artifact。

## 10. 插件工具

所有自定义工具必须以 `bounded_lite_` 开头，且匹配 `^[a-zA-Z0-9_-]+$`。

当前工具：

- `bounded_lite_route`
- `bounded_lite_plan_dag`
- `bounded_lite_plan_readiness`
- `bounded_lite_background`
- `bounded_lite_runtime_profile`
- `bounded_lite_model_config`

插件只做薄增强：路由解析、计划 DAG 校验、后台任务枚举、运行 profile 报告和角色模型配置。它不是第二套隐藏运行时。
