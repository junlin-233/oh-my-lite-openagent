多 Agent 开发框架 v1 架构图（最终版 · 2026-04-23）

  角色矩阵

  ┌────────────────────┬──────────┬──────────────────────────────────────────────────────────┐
  │       角色         │ 可见性   │                     能调用                                │
  ├────────────────────┼──────────┼──────────────────────────────────────────────────────────┤
  │ Command Lead       │ ✅ 可见  │ Task Lead, Plan Review, Result Review, Explore, Librarian │
  │ Plan Builder       │ ✅ 可见  │ Explore, Librarian, Plan Review (可选)                    │
  │ Deep Plan Builder  │ ✅ 可见  │ Explore, Librarian, Plan Review (强制)                    │
  │ Task Lead          │ 🔒 隐藏  │ Explore/Librarian (万不得已)                              │
  │ Explore            │ 🔒 隐藏  │ —                                                        │
  │ Librarian          │ 🔒 隐藏  │ —                                                        │
  │ Plan Review        │ 🔒 隐藏  │ Explore (只读)                                           │
  │ Result Review      │ 🔒 隐藏  │ Explore (只读)                                           │
  └────────────────────┴──────────┴──────────────────────────────────────────────────────────┘

  Plan Builder      = 强模型 + Review 可选
  Deep Plan Builder = 详细执行计划 + Review 强制（面向低强度执行模型交接）
  隐藏 Agent 的过程可通过 Ctrl+X+Down 展开查看（直接参考opencode的模式）

  入口 1：Command Lead（执行主入口）

  用户 ←→ Command Lead
           │
           ├─ 按需 Explore / Librarian（上游已调的不重复）
           │
           ├─ 简单 ──► 直接执行 ──► [用户可选] Result Review（只审 Command Lead 结果）
           │
           └─ 符合 Plan Builder 路由阈值 ──► 结构化 payload 传给 Plan Builder（子对话）
                      ┌──────────────────────────────────────┐
                      │ payload = 用户原话                   │
                      │        + 已收集 Explore/Librarian 片段│
                      │        + 约束列表                     │
                      │  （结构化，非摘要）                   │
                      └──────────────────────────────────────┘
                                │
                                ▼
                       Plan Builder 生成计划返回
                                │
                                ▼
                       Command Lead 解析计划 → 构建 Task DAG
                       ┌──────────────────────────────────────┐
                       │ 按 subtask.depends_on 推断依赖        │
                       │ 按 subtask.attributes 派遣模型        │
                       │   （映射表配置化，非硬编码）          │
                       │ 并发上限 3–5                          │
                       └──────────────────────────────────────┘
                                │
                                ▼
                       Task Lead 并行/串行执行
                                │
                                ▼
                       Command Lead 整体验收 ──► [用户可选] Result Review

  入口 2：Plan Builder（独立规划）

  用户 ←→ Plan Builder（强模型）（需要多轮对话明确用户需求，边界，不能自己想）
           │  按需 Explore / Librarian
           ▼
      生成计划 → 本地文件
           │
           ▼
      [可选] Plan Review
           │
           ▼
      用户新开对话 → Command Lead 读计划执行

  入口 3：Deep Plan Builder（详细执行计划 + 强制审查）

  用户 ←→ Deep Plan Builder（面向低强度执行模型的详细计划）（需要多轮对话明确用户需求，边界，不能自己想）
           │  按需 Explore / Librarian
           ▼
      生成计划 → 本地文件
           │
           ▼
      [强制] Plan Review  ←── 通过迭代补偿模型能力
           │
           ▼
      用户新开对话 → Command Lead 读计划执行

  Review 迭代规则（Plan / Result 统一）

  ┌──────────────────────────────────────────────────────────────────────┐
  │                                                                      │
  │   v1 ──► Review ─┬─ 通过 ──────────────────────► 结束                │
  │                  │                                                   │
  │                  └─ 不通过 + severity                                │
  │                         │                                            │
  │                         ├─ minor ──► 作者直接改，不再审              │
  │                         │                                            │
  │                         └─ major ──► 作者改 → Review                 │
  │                                      （总迭代硬顶 2 次，含重审）      │
  │                                                                      │
  │   v1 fail → v2 fail → v3 fail  ──────► escalate 给 Command Lead      │
  │                                                                      │
  ├──────────────────────────────────────────────────────────────────────┤
  │  severity 判定权在 Review 侧（防自利低估）                            │
  │  Review 反馈强制要求：                                                │
  │    · 定位到文件/行/函数                                               │
  │    · 每条反馈附可验证通过标准                                         │
  │    · 说不清楚怎么改 → 直接标 major，不让作者猜                        │
  └──────────────────────────────────────────────────────────────────────┘

    Plan Review   作用对象 = Plan Builder / Deep Plan Builder 的计划
    Result Review 作用对象 = Command Lead 的 execution-summary / 最终整合结果
    Result Review 不强制调用，由用户选择是否需要独立结果核查

  计划文件 Schema（必填字段）

  plan:
    subtasks:
      - id:            <unique>            # DAG 节点标识
        depends_on:    [id, ...]           # 必填，空数组也要写
        attributes:    [code, multimodal]  # tag set，驱动模型派遣
        deliverable:   <描述>               # Command Lead / 可选 Result Review 的验收依据
        description:   <做什么>

  信息流纪律

  ┌───────────────────────────────────────────────┐
  │ 上游 Explore/Librarian 结果                    │
  │             ↓ 打包进 payload                   │
  │ 下游优先消费 payload，判断是否还需补充          │
  │             ↓ 默认不补                         │
  │ Task Lead：除非万不得已，不主动 Explore/Librarian│
  └───────────────────────────────────────────────┘

    目的：避免同一信息在链路上被重复拉取

  标准派遣模板

  任何有委派权角色派给 subagent 的任务都必须显式写清：

    TASK
    EXPECTED OUTCOME
    ROLE
    SCOPE
    UPSTREAM EVIDENCE
    REQUIRED TOOLS
    MUST DO
    MUST NOT DO
    CONTEXT
    DELIVERABLE FORMAT
    FAILURE RETURN

  关键规则：

    - 不使用隐藏 initiator marker
    - SCOPE 必须限制文件/目录/模块/worktree/branch 或行为边界
    - MUST DO 必须包含维护本角色 todo、先消费 upstream evidence、验证结论
    - MUST NOT DO 必须包含不越界、不假设未检查现状、不做 whole-repo 无边界搜索
    - Task Lead 派遣必须包含 plan node id、depends_on、attributes、deliverable
    - Result Review 只能审 Command Lead 的 execution-summary / 最终整合结果

  Todo 纪律

  ┌───────────────────────────────────────────────┐
  │ 每个角色都维护自己的本地 todo 列表             │
  │ 多步工作开始先列 todo，执行中更新状态          │
  │ todo 只属于当前角色，不替代 artifact/canonical state │
  │ 不允许用 todo 形成新的隐藏控制平面             │
  └───────────────────────────────────────────────┘

  失败处理

  Task Lead 失败
      │
      └─► 写结构化状态返回
          ┌──────────────────────┐
          │ progress      进度    │
          │ blocker       阻塞点  │
          │ artifacts     已有产出│
          │ recoverability 可恢复性│
          └──────────────────────┘
                │
                ▼
          Command Lead 接手
                ├─ 独立叶子失败 ──► 记录，继续其他任务，最后汇总
                └─ 关键路径失败 ──► 停止下游，escalate 给用户

  设计原则（贯穿全框架）

  1. 相信模型质量 → 最小化审查，优先用契约约束而非多层保险
  2. 信息不重复拉取 → 上游结果沿 payload 向下传递
  3. 摘要是有损的 → 跨 agent 传递用结构化 payload，不用自然语言总结
  4. 审查权大于生成权 → severity 判定在 Review 侧，不在作者侧
  5. 失败优先 escalate，不暗自重试 → 避免隐藏问题累积
